# 工具执行调用栈与包装链 —— "为啥会跳?"

当 LLM 在它的回复里要求调用一个由扩展注册的工具时,agent-loop 会沿着这样一条调用栈真正把它跑起来:

```
9  execute(), tool-definition-wrapper.ts:17          ← 工具本体 execute 真在跑
8  execute(), wrapper.ts:24                          ← 扩展层再包一层,做 addedToolNames 检测
7  executePreparedToolCall(), agent-loop.ts:677      ← 入口"带着 signal 和 onUpdate 去调用 prepared.tool.execute"
6  匿名 (async thunk), agent-loop.ts:525            ← Promise.all 数组里的一个个 () => Promise<...>
5  匿名, agent-loop.ts:543                          
4  executeToolCallsParallel(), agent-loop.ts:543     ← "并行工具执行"调度
3  来自 await 的异步调用
2  executeToolCalls(), agent-loop.ts:427             ← 选择 "sequential 还是 parallel"
1  runLoop(), agent-loop.ts:214                      ← 主循环本次跑到了"执行工具"这一步
```

下文分两段:

- **第一段** 解释这条栈本身 —— 每一帧对应源码的哪一段、为什么有这条调用链。
- **第二段** 回答 "**为什么 `prepared.tool.execute` 会跳到 `wrapper.ts` 的 `execute`,而 `wrapper.ts` 的 `execute` 又会跳到 `tool-definition-wrapper.ts` 的 `execute`?这是什么机制?**" —— 重点讲 JS 的"包装链 + 闭包捕获"机制。

---

# Part 1 — 读懂这条栈

## 0. 栈帧怎么看

栈顶是当前正在执行的代码,往下每一帧是上一帧"被谁调起"的快照。"匿名"与"来自 await 的异步调用"不是普通函数,**是 JS 引擎为 anonymous 函数表达式和 `await` 微任务恢复点生成的隐式帧**,详见 §5。

读这种栈的标准姿势:

1. **从下往上** 顺着 "谁调了谁" 走。
2. **栈顶** 是你关心的"这一刻真正在跑的代码"——栈底是这条调用链最初被发起的地方。
3. **任何"匿名"或"异步调用"出现**,意味着对应那一帧在源代码里是个 `async (...) => { ... }` 或者 `await xxx`。

---

## 1. §1 §2 主循环挑了"并行"路径

### 1.1 `runLoop()` (agent-loop.ts:214)

主循环的某一轮,LLM 这次回了包含 tool calls 的 AssistantMessage。这一段真实的关键代码大致是:

```ts
const toolCalls = message.content.filter((c) => c.type === "toolCall");
if (toolCalls.length > 0) {
    const executedToolBatch =
        message.stopReason === "length"
            ? await failToolCallsFromTruncatedMessage(toolCalls, emit)   // 输出被截断 → 整批 fail
            : await executeToolCalls(currentContext, message, config, signal, emit);
    ...
}
```

含义:**主循环跑到"这一轮有工具调用"这一步**。

### 1.2 `executeToolCalls()` (agent-loop.ts:427)

```ts
async function executeToolCalls(currentContext, assistantMessage, config, signal, emit) {
    const toolCalls = assistantMessage.content.filter((c) => c.type === "toolCall");
    const hasSequentialToolCall = toolCalls.some(
        (tc) => currentContext.tools?.find((t) => t.name === tc.name)?.executionMode === "sequential",
    );
    if (config.toolExecution === "sequential" || hasSequentialToolCall) {
        return executeToolCallsSequential(...);
    }
    return executeToolCallsParallel(...);   // ← 截图的栈走到这里
}
```

含义:**这一批工具走"串行"还是"并行"?**默认走 `executeToolCallsParallel`。全局 `AgentLoopConfig.toolExecution = "sequential"` 或任何一个工具的 `executionMode: "sequential"` 才会退化为串行。

## 2. §3 §4 §5 §6 "并行执行"的 fan-out 模型

### 2.1 "来自 await 的异步调用"

V8 引擎为 `await` 加上的一帧隐式恢复帧。看到它,就说明这条调用链**曾经"暂停",等 microtask 再被唤醒**。

### 2.2 `executeToolCallsParallel()` (agent-loop.ts:543)

```ts
async function executeToolCallsParallel(...) {
    const finalizedCalls: FinalizedToolCallEntry[] = [];

    for (const toolCall of toolCalls) {
        await emit({ type: "tool_execution_start", ... });

        const preparation = await prepareToolCall(...);
        if (preparation.kind === "immediate") {
            // 已经知道要 fail 的工具,同步放进数组
            continue;
        }

        // ★ 关键:把"待执行"做成 async thunk 推入数组
        finalizedCalls.push(async () => {                            // ← §6
            const executed = await executePreparedToolCall(preparation, signal, emit);
            const finalized = await finalizeExecutedToolCall(...);
            await emitToolExecutionEnd(finalized, emit);
            return finalized;
        });
    }

    // ★ 关键:等所有 thunk 并行 resolve
    const orderedFinalizedCalls = await Promise.all(                 // ← §4
        finalizedCalls.map((entry) =>
            typeof entry === "function" ? entry() : Promise.resolve(entry),   // ← §5 map 回调
        ),
    );
    ...
}
```

含义:这是经典的**"先 fan-out,再 await Promise.all"**并行模式:

- **串行广播 `tool_execution_start` 事件** —— TUI/RPC 上要看到工具按"自然顺序"开始执行。
- **不立即调工具**,而是把每个工具包装成一个 `() => Promise<...>` thunk 推进数组。
- **最后一次性 invoke 所有 thunk** —— 多个互相不依赖的工具真的同时跑。
- 那些"在 prepare 阶段就已经 fail" 的工具是 `immediate` 类型,作为已 resolved Promise 同步进数组,不会产生异步 thunk。
- `Promise.all` 保证 **返回顺序与数组顺序一致**,方便后续按"原本 LLM 给的工具调用顺序"写 tool-result 消息,LLM 看到的上下文是确定的。

### 2.3 §6 "匿名 (async thunk), agent-loop.ts:525"

就是 `finalizedCalls.push(async () => { ... })` 这个被 push 进去的函数体。V8 看不到名字就标 "匿名"。

### 2.4 §5 "匿名, agent-loop.ts:543"

那个 `.map((entry) => typeof entry === "function" ? entry() : Promise.resolve(entry))` 的箭头函数调用 —— 也即 thunk **真正被 invoke** 那一行(`agent-loop.ts:543`)的回调帧。

> **§5 和 §6 对应同一个 thunk 体**,V8 给出两层堆栈:thunk 函数体本身和 map 回调里执行 thunk 的那一行。读到这里你心里可以画成:"这个工具的 execute 是被并行 fan-out 出来的 N 份之一"。

## 3. §7 实际调工具的入口

### 3.1 `executePreparedToolCall()` (agent-loop.ts:677)

```ts
async function executePreparedToolCall(prepared, signal, emit) {
    const updateEvents: Promise<void>[] = [];
    let acceptingUpdates = true;

    try {
        const result = await prepared.tool.execute(                  // ← §7
            prepared.toolCall.id,
            prepared.args as never,
            signal,
            (partialResult) => { ... },                             // → emit tool_execution_update
        );
        acceptingUpdates = false;
        await Promise.all(updateEvents);
        return { result, isError: false };
    } catch (error) {
        // 工具抛错 → 转成 isError: true 的 tool result,不带走整个 agent run
        ...
    }
}
```

含义:**这是给工具 4 个东西**:

1. `toolCall.id` —— 工具调用唯一 id。
2. `prepared.args` —— 已经过 schema 校验 + `prepareArguments` 适配后的参数。
3. `signal: AbortSignal` —— 工具要主动尊重它。
4. `onUpdate(partialResult)` —— 让工具流式输出"半成品结果",被转成 `tool_execution_update` 事件。
5. **异常吞噬**:工具抛错时,**立刻**把它转成 `isError: true` 的 tool result,**不让单个工具的崩溃带走整个 agent run**。

`prepared.tool.execute(...)` **就是从这里开始往下"跳"**。但是跳到哪里,跳多少层,取决于 `prepared.tool` 究竟是什么对象 —— 见 Part 2。

## 4. §8 §9 工具本体的两层包装

## 5. 这条栈告诉你什么"事实"

1. **这次跑的不是单工具,而是并行 fan-out 出来的一组工具里的一员**。同一时刻有 N 个一样的栈并行存在,只是这次只显示了其中一个。
2. **这个工具是"扩展注册的"**(因为 §8 `wrapper.ts:24` 这一层只对扩展工具生效)。如果跑的是内置工具(`bash`/`read`/`edit` 之类),这层不会出现在栈里。
3. **真正执行的代码**在栈顶 §9 `tool-definition-wrapper.ts:17` 那一行 —— 是你写扩展时 `execute: async (...) => { ... }` 入口的实际位置。
4. **栈里没有任何业务逻辑**,全是包装层 + 调度层。你写扩展时不需要关心并行、异常吞噬、abort、activeTools 检测 —— **agent-loop 已经替你做完了**,你只管"给定 id + params + signal + onUpdate,返回 AgentToolResult"。
5. **栈底一定停在 `runLoop()`**,因为 agent 内部任何调用最远起点都是主循环。

# Part 2 — "为啥会跳" 其实是普通函数调用

## 6. 一句话机制

**"包装"是注册时一次性发生的,不是调用时才拼。** `wrapRegisteredTool` 返回的那个 `AgentTool` 对象,它的 `.execute` 字段**已经是 layer B 的固定函数对象**;`wrapToolDefinition` 返回的 `.execute` 也已经是 layer C 的固定函数对象。所以 `prepared.tool` 指向的"工具"根本不是你写的那个原始工具,而是被裹了两层的产物。栈里那三个 `execute`,只是三个不同的函数对象恰好同名/同位置而已。

它们之间的"跳转",**就是普通的 JS 函数调用 + 闭包变量捕获**。

---

## 7. 把"create chain"和"call chain"分开

这两件事发生在完全不同时机:

| 时机 | 做什么 |
| --- | --- |
| **注册时**(启动 / 写扩展) | 三层"包装函数对象"被创建,相互持有闭包引用。最外层扔进 `AgentSession._toolRegistry` |
| **运行时**(LLM 调工具) | agent-loop 拿 `prepared.tool`,它**就是最外层**;沿字段读 + 函数调用推进去 |

**最关键的一件事**:`prepared.tool` 这个变量,**指向的是包装链的最外层对象**。它的 `.execute` 字段就是一个普通属性,指向 layer B 的函数对象。这就像 `obj.foo` 是 `() => bar()` 时,`obj.foo()` 就是在调那个 `() => bar()` 一样 —— **没有什么机制,就是一次字段读 + 一次函数调用**。

---

## 8. 三个 "execute" 分别是谁

记三个函数对象,从最内到最外叫 A / B / C。

### 8.1 LAYER A — 你写的扩展工具(下面随便举个例子)

```ts
pi.registerTool({
    name: "my_tool",
    execute: async (toolCallId, params, signal, onUpdate, ctx) => {
        // 你真正的工具逻辑
        return { content: [{ type: "text", text: "..." }], details: {} };
    },
});
```

### 8.2 LAYER C — tools/tool-definition-wrapper.ts:17

```ts
export function wrapToolDefinition(definition, ctxFactory?) {
    return {
        ...,
        execute: (toolCallId, params, signal, onUpdate) =>
            definition.execute(
                toolCallId,
                params,
                signal,
                onUpdate,
                ctxFactory?.() as ExtensionContext,    // ← 适配 5参 → 4参 + 懒构造 ctx
            ),
    };
}
```

C 的关键点:

- C 函数体里**没有 `execute` 这个变量名**,直接调 `definition.execute(...)`。
- `definition` 是被闭包捕获的 `ToolDefinition`(你 `registerTool` 时传进去的那个对象)。
- `definition.execute` 就是**你写的 LAYER A**。

### 8.3 LAYER B — extensions/wrapper.ts:24

```ts
export function wrapRegisteredTool(registeredTool, runner) {
    const tool = wrapToolDefinition(registeredTool.definition, () => runner.createContext());
    const execute = tool.execute;                  // ← 闭包变量,捕获到 LAYER C 的函数对象
    return {
        ...tool,
        execute: async (toolCallId, params, signal, onUpdate) => {     // ← 这就是栈顶 wrapper.ts:24
            const activeBefore = runner.getActiveTools();
            const result = await execute(toolCallId, params, signal, onUpdate);  // → 调到 LAYER C
            const activeAfter = runner.getActiveTools();
            ...
        },
    };
}
```

B 的关键点:

- `const execute = tool.execute` 把 layer C 的函数对象**快照**到一个闭包变量。
- B 自己的函数体里 `await execute(...)`,**调的就是 layer C**。
- B 就是栈顶 wrapper.ts:24 那帧;那行 `result = await execute(...)` 才是真正"跳进 layer C"的入口。

**最关键的一句**:`prepared.tool` 指向的是 `wrapRegisteredTool` 的返回对象,它身上的 `.execute` 就是 LAYER B。**所以 `prepared.tool.execute(...)` 一开始就在调 B**。

## 9. 把动作连起来看

### 9.1 注册时(启动一次就够)

```text
wrapRegisteredTool(myExtTool, runner)
   │
   ├─ wrapToolDefinition(...)
   │     └─ closure captured: definition = myExtTool.definition
   │     return { ... execute: (LAYER C) }     ← 把 LAYER C 挂在返回对象的 .execute 上
   │
   ├─ const execute = tool.execute              ← 闭包变量 execute = LAYER C 的函数对象
   │
   └─ return { execute: (LAYER B) }             ← 把 LAYER B 挂在返回对象的 .execute 上
                                                          ↑
                                                          └─ _toolRegistry["my_tool"] = 这个最外层包装
```

### 9.2 运行时(LLM 一调工具)

```text
prepared.tool.execute(...)
   ↑ 这个 .execute 字段就是 LAYER B 的函数对象
   ↓
LAYER B 函数体:
   activeBefore = runner.getActiveTools()
   result = await execute(...)            ← execute 是闭包变量,指向 LAYER C
   ↓
LAYER C 函数体:
   ctx = ctxFactory?.()                   ← 闭包变量 ctxFactory = () => runner.createContext()
   return definition.execute(id, params, signal, onUpdate, ctx)
   ↑ definition 是 closure = myExtTool.definition → 调用你写的 LAYER A
   ↓
LAYER A: 你的工具代码跑起来
```

## 11. 洋葱模型类比

把 "prepared.tool" 想成一颗洋葱,从外到内一共三层:

```
┌────────────────────────────────────────┐
│ LAYER B (wrapper.ts:24)                │  ← 加 addedToolNames 监测(getActiveTools 前后差集)
│  ┌──────────────────────────────────┐  │
│  │ LAYER C (tool-definition-wrapper.ts:17) │ ← 5参 → 4参 适配,懒构造 ctx
│  │  ┌────────────────────────────┐  │  │
│  │  │ LAYER A (你写的 execute)        │  │  ← 真正干活的代码
│  │  └────────────────────────────┘  │  │
│  └──────────────────────────────────┘  │
└────────────────────────────────────────┘
```

每次你"戳一下外层"(B 的 `.execute`):
1. B 完成自己的事(前后两次 `getActiveTools()`);
2. 通过闭包变量 `execute` 戳中间层(C 的 `.execute`);
3. C 完成自己的事(5 → 4 参数适配 + `ctx` 懒构造);
4. 通过闭包变量 `definition.execute` 戳最里层(A);
5. A 开始干你写的活。

每一次"戳",就是一次普通的 `obj.execute(...)` 函数调用。**没有任何"机制"把多个函数粘在一起**。

## 12. 最小复现验证

```ts
// === LAYER A: 你写的工具 ===
const layerA = {
    name: "greet",
    execute: async (id, params, signal, onUpdate, ctx) => {
        console.log("LAYER A 实际跑:", params);
        return { content: [{ type: "text", text: `hello, ${params.name}` }], details: {} };
    },
};

// === LAYER C: 5 参 → 4 参适配 ===
function wrapToolDefinition(definition) {
    return {
        ...definition,
        execute: (id, params, signal, onUpdate) => {
            console.log("LAYER C 适配 + 懒构造 ctx");
            const ctx = { mode: "fake" };
            return definition.execute(id, params, signal, onUpdate, ctx);  // → 跳进 A
        },
    };
}

// === LAYER B: 检测 addedToolNames ===
function wrapRegisteredTool(registeredTool, runner) {
    const inner = wrapToolDefinition(registeredTool);   // ← 拿到 LAYER C
    const execute = inner.execute;                     // ← 闭包变量 = LAYER C
    return {
        ...inner,
        execute: async (id, params, signal, onUpdate) => {
            console.log("LAYER B addedToolNames 检测");
            const result = await execute(id, params, signal, onUpdate);  // → 跳进 C
            console.log("LAYER B 拿到 result:", JSON.stringify(result));
            return result;
        },
    };
}

// === 准备好的 "prepared.tool" — 每次 runtime 拿到的就是这个对象 ===
const wrapped = wrapRegisteredTool(layerA, null);
//   ↑ wrapped.execute 就是 LAYER B
//   ↑ wrapped.execute 内部有个闭包变量 execute,指向 LAYER C

// === 调用 ===
await wrapped.execute("tool-1", { name: "alice" }, undefined, undefined);
```

跑起来输出:

```
LAYER B addedToolNames 检测
LAYER C 适配 + 懒构造 ctx
LAYER A 实际跑: { name: 'alice' }
LAYER B 拿到 result: {"content":[{"type":"text","text":"hello, alice"}],"details":{}}
```

跟你的真实栈一一对应:

- 栈顶 (`wrapper.ts:24`) → 第一行 log → 进入 LAYER B
- 第二行 log → LAYER B 里 `await execute(...)` 推进到 LAYER C
- 第三行 log → LAYER C 里 `definition.execute(...)` 推进到 LAYER A

没有"机制",只有普普通通的 `func(args)`。

## 13. 记住三件事就够了

1. **`prepared.tool` = `AgentSession._toolRegistry[name]` = `wrapRegisteredTool` 的返回值**。它的 `.execute` 是 LAYER B 的函数对象,**注册那一刻,包装链就冻结了**。
2. **每次 `await prepared.tool.execute(...)` 就是在调 LAYER B**;LAYER B 通过闭包变量 `execute` 调到 LAYER C;LAYER C 通过闭包变量 `definition.execute` 调到你写的 LAYER A。链上的每一"跳"都是普普通通的 `func(args)` 语法。
3. **LAYER B 和 C 的"添加行为"全靠普通变量**:B 多了"前后两次 `getActiveTools()` 取差集",C 多了"懒构造 ctx"。**它们没有用 Proxy、没有用 `defineProperty`、没有任何隐式机制** —— 全部都是普通函数体里手写的语句。

如果你想亲手验证:在 `extensions/wrapper.ts:24` 那行打个 debugger,或者在 B / C / A 各自加一行 `console.log("layer X")`,然后用 LLM 调一下这个工具 —— 你会得到和上面迷你复现完全一样的观察结果。

## 附:为什么这次栈里有 §8 (wrapper.ts:24)、但内置工具的栈里没有?

只有**扩展注册的工具**会过 `wrapRegisteredTool`,因为 `AgentSession._refreshToolRegistry()` 里有:

```ts
const wrappedExtensionTools = wrapRegisteredTools(allCustomTools, runner);
const wrappedBuiltInTools  = wrapRegisteredTools(allBaseToolDefinitions, runner);   // 内置工具也走 wrapper!
```

> 内置工具(base tools:`bash`/`read`/`edit`/`write`/`grep`/`find`/`ls`)也走 `wrapRegisteredTool`,所以它们的栈同样会有 `wrapper.ts:24` 这一层。
>
> 真正"拿掉 §8"的是 `AgentLoopConfig.toolExecution === "sequential"`,或者工具执行走了 `_installAgentToolHooks` 之后被 `beforeToolCall` 阻断,但栈帧合成是分开的(工具压根没跑)。
>
> 还有一种特殊情况:`createToolDefinitionFromAgentTool(tool)` 把一个**纯 AgentTool**(无 ctx 工厂)反向包成 ToolDefinition 时,产生的是"单层 LAYER C + 直接 LAYER A",**没有 B 层** —— 用于 SDK 调用方传入 `baseToolsOverride: { name: AgentTool }` 的场景。

如果你看到一条几乎一样的栈但**少了 §8 那一帧**,那就是这条路径在跑;否则就是默认的"两层包装 + 你写的 execute"。
